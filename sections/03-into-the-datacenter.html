Into the Datacenter

We left your request as a stream of encrypted packets, addressed to api.anthropic.com, disappearing into the network. Now they arrive. The first machines to touch them are edge servers whose job is mundane but essential: they hold the server's private key and complete the TLS handshake, so this is where the encryption is finally peeled away and your request becomes readable JSON again — at the far end of the encrypted tunnel, on infrastructure operated by or for Anthropic, and nowhere else along the way.

The decrypted request then meets an API gateway. Before any model sees a single byte, the gateway authenticates the API key in the x-api-key header, checks it against rate limits and quotas, and validates the JSON body — is the model name real, is max_tokens sane, does the message list parse? Requests that fail any of these checks are turned away right here, with the 401, 429, or 400 answered in a few milliseconds without waking a single GPU. Requests that pass are handed to a load balancer, which picks one backend out of a large fleet of model servers — and because each fleet holds one model's weights, requests for Opus and for Haiku part ways here, traveling to different pools of machines.

One honest caveat: the exact layout of Anthropic's production stack is not public, and we won't pretend to know it. Everything here describes how large-scale LLM inference services are generally built — the standard, well-documented machinery of the field. The real system surely differs in details, but these are the mechanisms any service at this scale relies on.

A model too big for one chip

Why does the request need a special "model server" at all, rather than any machine in the building? Because a frontier model is mostly one enormous block of numbers: its weights run to hundreds of gigabytes, far more than any single GPU (or other accelerator; we will use the terms interchangeably) can hold. So the model is sharded — sliced across a group of accelerators wired together with fast interconnects, and that whole group acts as one logical server. The single box labeled "model server" above really contains several chips, each holding its slice of the weights, all working in concert on every token generated for you.

Those weights were loaded once, when the server started, and stay resident in accelerator memory permanently. This inverts the usual picture of a web service: here the model is far too heavy to move, so it never does: your few kilobytes of request travel to the weights, flow through them, and leave. The weights are the factory; your request is the workpiece.

Sharing the assembly line

One fact shapes how these servers are run: an accelerator performs the arithmetic for one generation step so quickly that serving a single user would leave it idle almost all the time. The expensive part of each step is streaming those hundreds of gigabytes of weights through the compute units — and having streamed them, the chip can apply them to one request or to dozens for nearly the same cost. So serving systems batch: many strangers' requests ride through the very same forward passes, sharing each read of the weights.

The refinement that makes this practical is called continuous batching. Requests don't wait for a full busload to assemble; they join and leave the batch at the granularity of a single token. The moment one request frees its slot, another slides in mid-stream, and the assembly line never pauses. The figure below shows one server's batch as a timeline: each row is a slot, each column one forward pass, and each rounded bar one request — a short bright chunk when its prompt is absorbed, then one small cell per generated token. Notice what the readout implies: the server's total output grows almost linearly with occupancy, while each individual request still gets a token on every pass — which is why batching is universal, multiplying the useful work per pass without slowing any one user much.

Prefill and decode

Within each request, the work splits into two very different phases, and the split explains something you have felt every time you use Claude. First comes prefill: the entire prompt is pushed through the model in one massively parallel pass, because all its tokens are already known and can be processed at once. This phase is limited by raw arithmetic, and its duration is your time to first token. Then comes decode: tokens are generated one at a time, each requiring its own complete pass through every layer, limited not by arithmetic but by how fast the weights can be streamed from memory. One big gulp, then many small sips — that is the pause-then-stream rhythm you know. In the demonstration below, you can replay both phases in slow motion, and lengthen the prompt to see the first-token wait grow while the decode rate stays flat.

Prefill is also where a well-known optimization lives. If the beginning of your prompt is identical to one the server processed moments ago — and in an agent loop it almost always is, since every iteration resends the same system prompt, tool definitions, and conversation so far — the service can keep the computed internal state of that prefix and skip re-prefilling it. This is prompt caching — unlike much of this section, a documented feature of Anthropic's API — and Claude Code marks the stable prefixes of its requests as cacheable for exactly this reason — the difference between re-reading the whole transcript from scratch on every turn and paying only for what's new.

There is one thing we have quietly glossed over. We keep saying the model processes "tokens," but your request is still text — bytes of UTF-8. Before the first layer of the model can touch it, that text has to become numbers. How that happens is our next stop.
