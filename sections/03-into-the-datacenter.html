A model too big for one chip

A frontier model is mostly one enormous block of numbers: its weights run to hundreds of gigabytes, more than any single GPU (or accelerator) can hold. So the model is sharded, sliced across a group of accelerators wired together by fast interconnects, and the whole group acts as one logical server. The diagram below unpacks that single "model server" box: it holds several chips, each with one slice, all working in concert on every token.

A model too big for one chip

A frontier model's weights run to hundreds of gigabytes, far beyond the tens of gigabytes of high-bandwidth memory on one accelerator, so the model is sharded across chips that act as one logical server, cut along two axes. Tensor parallelism splits individual weight matrices: each chip holds a vertical slice of every layer and computes its partial for each token, and the partials are combined by an all-reduce before the next layer. That all-reduce fires at essentially every layer, so it is tolerable only over a very fast interconnect — NVLink-class links within a node, not ordinary Ethernet. Pipeline parallelism instead splits by depth: each chip holds a block of consecutive layers and tokens flow down the pipeline. Its cross-chip traffic is far smaller — only activations at block boundaries — so it tolerates slower links and spans nodes, at the cost of a "bubble" of idle chips that in-flight micro-batches must hide. Real deployments combine the two. Anthropic's exact configuration is not public, but the constraint — all-reduce cost against interconnect bandwidth — is universal. Below, the box opens to several chips, each with one tensor-parallel slice; double arrows mark the all-reduce within a node, the single arrow the pipeline split across nodes.

Sharing the assembly line

One fact shapes how these servers run: an accelerator finishes one generation step so fast that serving a single user leaves it idle almost all the time. The costly part is streaming hundreds of gigabytes of weights through the compute units — and once streamed, they apply to one request or to dozens for nearly the same cost. So servers batch, riding many strangers' requests through the same forward passes.

The refinement that makes this practical is continuous batching: requests join and leave at single-token granularity, so the moment one frees its slot another slides in mid-stream. Sharing the assembly line

The naive form, static batching, collects a fixed group and runs it until all finish — wasteful, since requests finish at different lengths and the batch stalls on its slowest member while freed slots idle. Continuous batching (also in-flight batching, from the Orca work) manages the batch at single-token granularity: after each forward pass the scheduler evicts any request that emitted its stop token and admits a waiting one, so a newcomer's prefill interleaves with others' decode and no slot idles. Prefill and decode

Within each request the work splits into two very different phases. Prefill pushes the entire prompt through the model in one massively parallel pass, since all its tokens are known at once; limited by raw arithmetic, its duration is your time to first token. Then decode generates tokens one at a time, each a full pass through every layer, limited not by arithmetic but by how fast the weights stream from memory. Prefill and decode

Within a request the work divides into two phases of opposite character. Prefill pushes all prompt tokens through the model in one pass; since every token is known, the layer operations are large matrix-matrix multiplies with high arithmetic intensity, so prefill is compute-bound, its cost scaling with prompt length — this is your time to first token. Decode is the opposite: each new token needs a full pass through every layer for one token alone, collapsing those multiplies into skinny matrix-vector products with almost no arithmetic reuse, so the bottleneck becomes memory bandwidth — one token means streaming the entire weight set out of high-bandwidth memory once. That gives a hard ceiling: if a replica holds N bytes of weights and its HBM delivers B bytes per second, a single stream cannot exceed roughly B/N tokens per second. Put numbers on it: 150 GB of weights over an aggregate 5 TB/s tops out near 33 tokens per second per stream. Batching pays because that one weight-streaming serves the whole batch at once. One last trick: as you work with Claude, the start of each new message often repeats the last, so rather than re-read it the kitchen keeps its notes and reads only what is new. Prefill is also where a well-known optimization lives. If the start of your prompt matches one processed moments ago — and in an agent loop it almost always does, since every turn resends the same system prompt, tool definitions, and transcript — the service reuses that prefix's computed state and skips re-prefilling it. This is prompt caching, a documented feature of Anthropic's API, and Claude Code marks its stable prefixes as cacheable for that reason.
