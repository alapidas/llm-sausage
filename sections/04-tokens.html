Tokens

To the computer, your message is just text, but the model does not think in letters — it thinks in numbers. So your text is chopped into small pieces called tokens, each swapped for a number. A token is a chunk of text: a short word like the, part of a longer word, or a single mark. Nobody chose these pieces; the model learned them by reading a huge pile of text and repeatedly gluing the most common pair of neighbors together. A bare number means nothing alone, so the model keeps a giant table with one row of numbers per token, standing for its meaning — and words about similar things end up with similar rows, as if placed near each other on a map nobody arranged. Where the vocabulary comes from

Nobody chooses those fragments by hand; they are learned by byte-pair encoding, a procedure that is almost embarrassingly simple. Start with the 256 possible bytes, so every text is a sequence of single-byte tokens, then scan a huge corpus for the most frequent adjacent pair — in English, roughly t then h — fuse it into a new token, and rewrite the corpus. Repeat a hundred thousand times: frequent sequences condense early, rare ones never do. In the demonstration below you can watch this on a tiny corpus, one merge at a time, as th, the, and then ␣the crystallize out of the character soup:

The vocabulary is the output of byte-pair encoding. Seed it with the 256 byte values, express the corpus as single-byte tokens, then repeatedly find the most frequent adjacent pair and add it as a new token, recording the order each was learned. That order is the merge rank, essentially the whole tokenizer at inference time: to encode new text you apply merges greedily, always taking the lowest-ranked applicable merge first and replaying until no ranked pair remains. Encoding is therefore deterministic and roughly linear in length, and whatever was common in training stays cheap forever after.

Two consequences follow. First, the 256-byte floor never leaves the vocabulary, so byte fallback is always available: an unseen emoji, a rare script, or arbitrary binary junk decomposes into its UTF-8 bytes rather than failing — there is no unknown-token case, only inputs that cost more tokens. Second, vocabulary size is a genuine tradeoff: a larger vocabulary packs more text into fewer tokens, shortening sequences and saving compute in the length-bound layers, but it inflates the embedding and output-projection matrices and spends parameters on rare merges seen little in training. A hundred thousand entries is a common settling point.

From ids to vectors

So the tokenizer hands the model a list of integers, and an integer is barren: id 4,072 is no more similar to 4,073 than to 90,000. The model proper first exchanges each id for something richer, from a giant table — the embedding matrix — one row per vocabulary entry, each a vector of thousands of numbers; looking up a token is just reading row id. Everything the model knows about a token in isolation lives in that vector, encoded geometrically: related meanings sit near each other and consistent directions capture consistent relationships. Nobody designed this — the model is graded endlessly on next-token prediction, and nudging related tokens toward similar vectors improves those predictions, so the geometry emerges. In the demonstration below you can tap the words on the map to select a token and inspect a schematic slice of its vector, watching the neighborhoods organize themselves and nearby words carry similar-looking columns of values:

The tokenizer's output is a list of integer ids, ordinally meaningless — 4,072 and 4,073 are neighbors only by the accident of merge order. The first parametric operation replaces each id with a learned vector by indexing an embedding matrix of shape V × d_model: one row per vocabulary entry, each a d_model-dimensional vector. The lookup is a gather — row id read straight from the table, no matrix multiply required.

The arithmetic is worth doing explicitly. With V near 100,000 and d_model several thousand, the embedding matrix alone runs to 100,000 × 4,000 = 400 million parameters — hundreds of millions of weights before a single attention head exists. Many models tie weights: the same matrix, transposed, serves as the output projection that turns the final hidden state back into a distribution over the vocabulary, amortizing that table across both ends and keeping input and output vocabularies consistent by construction.

Each row holds learned geometry, not design: graded only on next-token prediction, the network finds that arranging related tokens along nearby directions lowers the loss, so semantic neighborhoods and relational directions emerge on their own. The accounting lives here too — because every id is one row and the sequence of ids is what the model consumes and the meter counts, tokens, not characters or words, are the unit of both context window and price. Transliterated Klingon costs more than equal-length English because it earned fewer merges, and asking a model to count a word's letters asks about characters it never received.
