Tokens

The request has cleared the routing layers and reached a machine that holds the model's weights. What it carries is still just text — your prompt, the conversation so far, the tool definitions, all of it one long string of bytes. A neural network cannot do anything with a string of bytes directly. It computes with numbers, so before anything else happens the text is translated: it is cut into tokens, and each token is exchanged for an integer.

A token is not a character, and it is not quite a word. It is an entry in a fixed vocabulary of text fragments — modern models use vocabularies on the order of a hundred thousand entries or more. Common words are single tokens. So are common prefixes and endings, frequent letter pairs, punctuation runs, and — perhaps surprisingly — words with a leading space attached: the and ␣the are two different entries, because inside running text a word almost always arrives with the space that precedes it. Rare words don't get an entry of their own; they are assembled from several smaller pieces. In the demonstration below you can type any text and watch it split. Each colored pill is one token, with its integer id underneath, and the ␣ marks show where a space belongs to the token itself:

A few things are worth noticing as you play with it. The code preset tokenizes remarkably economically: function, return, a run of indentation, the (); at the end of a call — each is a single token, because tokenizers are trained on enormous amounts of source code and these sequences occur constantly. A long rare word shatters into many small fragments. Numbers shatter too, often digit by digit, which is one honest reason arithmetic is not a model's most natural skill. And the emoji becomes raw bytes: the vocabulary always keeps the 256 individual byte values as a floor, so any input whatsoever — any language, any binary junk — can be represented. There is no such thing as an unknown word, only a word that costs more tokens.

Where the vocabulary comes from

Nobody sits down and chooses those hundred thousand fragments by hand. They are learned, with a procedure called byte-pair encoding that is almost embarrassingly simple. Start with a vocabulary of just the 256 possible bytes, so every text is a long sequence of single-byte tokens. Now scan a huge corpus of text and find the pair of adjacent tokens that occurs most often — in English that is something like t followed by h. Fuse that pair into a new token, add it to the vocabulary, and rewrite the corpus using it. Then repeat: find the most frequent adjacent pair in the rewritten corpus, fuse it, repeat again, on the order of a hundred thousand times. Frequent sequences condense into single tokens early; rare ones never do. This training history is exactly why tokenization has the texture you saw in the first demonstration. The merges are ranked by frequency, and encoding new text just replays them in order, so whatever was common in the training corpus is cheap forever after. It also explains the sharp edges. Hello and ␣hello are unrelated entries, so capitalization and spacing genuinely change what the model receives. Your API bill and the context window are both measured in tokens, so the same program is cheaper to send than the same number of characters of, say, transliterated Klingon. And when you ask a model how many letters are in a word, you are asking about characters it never actually saw — it received one or two opaque ids, and any spelling knowledge it has was absorbed indirectly during training rather than read off the input.

From ids to vectors

So the tokenizer hands the model a list of integers. An integer, though, is a barren thing: id 4,072 is not "more similar" to id 4,073 than to id 90,000. The very first thing the model proper does is exchange each id for something far richer. It holds a giant table — the embedding matrix — with one row per vocabulary entry, and each row is a vector of thousands of numbers. Looking up a token is nothing more than reading row id of that table. Everything the model can be said to know about a token in isolation lives in that vector, and the knowledge is encoded geometrically: tokens with related meanings sit near each other, and consistent directions in the space capture consistent relationships. Nobody designed this arrangement. During training the model is endlessly graded on predicting the next token, and nudging related tokens toward similar vectors happens to make those predictions better, so the geometry simply emerges. Anthropic does not publish the production tokenizer used by its models, so the tokenizations in this section come from a small illustrative byte-pair tokenizer with a few hundred merges built for these figures. The behavior it exhibits — greedy merges, whitespace sensitivity, rare words shattering, bytes as a fallback — is faithful to how real tokenizers work, but the exact token boundaries and ids will differ from what the real vocabulary produces.

Your entire request is now a sequence of vectors, thousands of numbers per token, lined up in order. That sequence is the actual input to the transformer, and pushing it through — the forward pass — is where we go next.
