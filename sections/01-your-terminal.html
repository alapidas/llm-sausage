Your Terminal

It starts with a sentence. You open a terminal, run claude, and ask for something ordinary — fix the failing test, rename this function. You press Enter, a spinner turns, and an answer streams back character by character, as if someone were typing. It feels like a chat window. It is nothing like a chat window. Between your keystroke and the first character of the reply, your sentence will be folded into a document hundreds of thousands of characters long, serialized into JSON, carried across the internet to a datacenter, sliced into tokens, pushed through layer after layer of matrix arithmetic on GPUs, and turned back into text one small piece at a time. We will follow that request the whole way there and back, beginning on your own machine — where, before any AI is involved, a surprising amount is already going on.

A program drawing a picture of a prompt

The interface that greets you is not the terminal’s own prompt. It is a JavaScript program — installed via npm it runs on Node.js; the native installer bundles its own runtime — that has taken over the window and is drawing a text-mode user interface, somewhat the way a game draws frames. Ordinarily a terminal runs in what Unix calls canonical mode: the operating system collects your keystrokes into a line buffer, handles backspace for you, echoes each character to the screen, and only hands the finished line to the program when you press Enter. An interactive interface cannot work that way — it wants to react to every keystroke, offer completions, move a selection with the arrow keys. So Claude Code switches the terminal into raw mode. From then on every key you press is delivered to the CLI process immediately, as one or a few raw bytes on standard input, and nothing is echoed automatically. If the program did nothing, your typing would be invisible.

Every character you see is therefore the program rendering, not the terminal echoing. When a byte arrives, the CLI updates its internal state — the text of your draft message, the cursor position — and repaints the affected part of the screen by writing ANSI escape sequences to standard output: move the cursor here, switch to this color, erase to the end of the line, write these characters. The blinking block cursor, the border around the input box, the spinner that appears while a request is in flight — all of it is redrawn, many times a second, through the same output channel your shell uses to print the results of ls. What Enter actually sends

When you press Enter, it would be reasonable to assume the CLI sends your sentence to the model. It does — but your sentence rides along at the very end of something much larger. The API is stateless: between calls, the model retains no memory of your project or your earlier messages. So on every single request, the CLI assembles the full context from scratch. First comes a large system prompt: several thousand words of instructions about how to behave — how to format answers, when to ask before acting, how to use tools. Then the tool definitions: every tool the model may request — Bash, Read, Edit, Grep, and more — each described as a JSON schema, its parameters spelled out in full. Then project memory: files such as CLAUDE.md, notes about how this codebase works. Then the entire conversation so far: every message, every reply, and every tool result — file contents, command output, search matches — accumulated over the session. And then, at last, your new message.

The figure below lays this assembly out as a single horizontal bar: the model’s context window, with room for 200,000 tokens. A token is a short chunk of text, roughly three-quarters of an English word; we will meet tokens properly in a later section. Drag the slider to extend the conversation, and hover over or touch each segment to see what it holds. Notice how quickly the history grows — each turn adds not only what was said but everything the tools returned — and how thin the green sliver of your new message remains throughout.

The segment sizes here are illustrative rather than measured; they change between versions. But the system prompt and tool schemas together typically account for tens of thousands of tokens before you have typed anything at all.

An agent, not an oracle

There is a second surprise hiding in that bar: the tool results. Where do they come from? The model itself cannot open your files, run your tests, or execute anything. It receives text and produces text — that is the entirety of its powers. What makes Claude Code an agent is the program wrapped around the model. When the model wants to look at auth.spec.ts, it replies not with an answer but with a structured tool_use request — in effect, “please run Read on this path.” The CLI checks that request against your permission settings (asking you first when the rules require it), executes the tool locally on your machine, packages the output into a tool_result message, appends it to the conversation, and sends the whole context to the API again. The model, now seeing the file’s contents, may ask to run the test suite — and the cycle repeats.

In the animation below you can watch this loop turn. A single “fix the failing test” routinely fans out into half a dozen API calls before the model finally responds with plain prose instead of another tool request. How a finished reply becomes an edit on your disk is a story for the final section.

Step back and notice what this architecture implies. Everything the model will ever know about your request is inside that one growing document. It cannot peek at your screen, browse your filesystem in the background, or recall what you asked yesterday. If a fact is not in the context — placed there by the system prompt, a memory file, or a tool result the model itself asked for — then for the model it does not exist. There is no hidden channel. The context is the interface.
