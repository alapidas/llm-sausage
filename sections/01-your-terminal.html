Your Terminal

Claude Code lives in a terminal — a plain text window where you type commands instead of clicking. Unlike an ordinary terminal, which waits for Enter before showing your letters, it reacts to every keystroke, redrawing the screen many times a second. When you press Enter, your sentence does not travel alone: the model keeps no memory between messages, so each time it is handed the whole story — standing instructions, the actions it may take, notes about your project, and the conversation so far — with your sentence as the last line. And because it can only read and write text, whenever it wants to open a file or run a command it must ask the program, which does the work and hands back the result. Everything the model knows lives on that one growing page. Next, that page leaves your computer.

It starts with a sentence: you run claude, ask it to fix a failing test, and an answer streams back. The interface is not the terminal’s own prompt but a program drawing text — it switches the terminal into raw mode so every keystroke reaches it at once, then repaints with ANSI escape sequences on the same channel your shell uses to print ls. The API is stateless, so the CLI rebuilds the whole context each call: a system prompt, the tool schemas (Bash, Read, Edit…), project memory, the entire conversation, and finally your message. The model cannot touch your machine; when it wants a file it emits a tool_use request, the CLI runs it and returns a tool_result, and the loop turns until the model answers in prose. A program drawing a picture of a prompt

The interface greeting you is not the terminal’s own prompt but a JavaScript program drawing a text-mode interface. Normally a terminal runs in what Unix calls canonical mode: the operating system buffers your keystrokes into a line, handles backspace, echoes each character, and hands over the finished line only at Enter. An interactive interface cannot work that way, so Claude Code switches into raw mode, where every key reaches the CLI process immediately as a raw byte or two, nothing echoed automatically.

Every character you see is thus the program rendering, not the terminal echoing. When a byte arrives, the CLI updates its state and repaints the affected region by writing ANSI escape sequences to standard output. All of it is redrawn many times a second, on the same channel your shell uses to print ls. The window is not the terminal’s prompt but a program that has seized control of it (npm-distributed for Node.js, or shipped with its own runtime). A Unix terminal’s driver defaults to canonical mode, governed by a termios struct whose flags set the line discipline: ICANON buffers bytes until a newline before waking the reader, and ECHO prints each byte back. A text UI cannot wait for newlines, so the CLI switches the descriptor to raw mode: it clears ICANON and ECHO, disables signal generation, and sets VMIN = 1, VTIME = 0 so read() returns on the first available byte, not a whole line. Keystrokes then arrive at once on standard input, echoed only if the program echoes them.

Most keys arrive as the obvious byte; the interesting ones do not. The up arrow sends three bytes — ESC, [, A, that is \x1b[A — a CSI (Control Sequence Introducer) escape sequence; Home, End, the function keys, and modified chords each encode their own, which the program parses to recover the keypress. Output uses the same alphabet in reverse — \x1b[H homes the cursor, \x1b[2K erases a line, \x1b[38;2;…m sets a truecolor foreground — on the very channel your shell uses to print ls.

Built on a React renderer for the terminal (Ink), the program keeps a virtual screen and re-renders it like frames: each state change diffs against the last and emits only the escape sequences to reconcile them. On resize the kernel delivers SIGWINCH; the process re-reads rows and columns through the TIOCGWINSZ ioctl and repaints. What Enter actually sends

Your sentence goes at the very end of something much larger. The API is stateless: the model retains no memory between calls, so the CLI rebuilds the context from scratch on every request. First a large system prompt: several thousand words on behavior and tool use. Then the tool definitions — Bash, Read, Edit, Grep, and more, each as a JSON schema. Then project memory such as CLAUDE.md. Then the whole conversation so far — every message, reply, and tool result. And finally your new message.

Your sentence rides at the end of a document the CLI rebuilds from nothing on every call: the Messages API is stateless, holding no session between requests. Your new message is appended last.

History dominates: every turn adds not just what was said but every byte the tools returned — whole files, command output, search hits — all resent verbatim on the next call. An agent, not an oracle

Where do those tool results come from? The model cannot open your files or run your tests; it only receives and produces text. What makes Claude Code an agent is the program wrapped around it. When the model wants auth.spec.ts, it replies not with an answer but with a tool_use request. The CLI checks it against your permission settings, prompting you when required, runs the tool locally, packages the output into a tool_result message, and sends the whole context back. Seeing the file, it may ask to run the test suite, and the cycle repeats.

In the animation below you can watch this loop turn: a single “fix the failing test” routinely fans out into half a dozen API calls before the model answers with plain prose instead of another tool request. How that reply becomes an edit on your disk is the final section’s story.

Those tool results expose the architecture. The model is a pure text-to-text function: context in, tokens out — it cannot open a file or start a process. The agent is the loop the CLI wraps around it. When the model needs auth.spec.ts, it does not answer; it emits an assistant turn carrying a tool_use block and a stop reason of tool_use rather than a finished reply. The CLI checks the call against your permission configuration, prompting you when the rules demand, runs the tool locally, wraps the output in a tool_result block, and issues a fresh request with the enlarged context. Able now to read the file, it may emit another tool_use, and the wheel turns again.
