The Way Back

The model has committed to a token. Right now it is nothing but an integer in accelerator memory — say, id 8543 — chosen by the sampler at the end of a forward pass. For that number to become a flicker of text in your terminal, it has to make the entire journey of this article in reverse, and it starts immediately. The serving stack does not wait for a sentence, or even a word, to finish. The id is looked up in the tokenizer's vocabulary, mapped back to its bytes — perhaps Fix — and pushed onto the HTTPS connection that has been sitting open this whole time. This is the connection the CLI asked for back when it set "stream": true in the request body: instead of one response, the server answers with Content-Type: text/event-stream and keeps the socket alive, trickling the reply out as server-sent events, one small encrypted chunk at a time, back through the same TLS tunnel the request came in on.

The stream has a fixed grammar. First comes a message_start event carrying the shell of the reply — its id, the model name, and how many input tokens the prompt consumed. Then, for each block of content, a content_block_start, followed by many content_block_delta events, each holding a small fragment of text. A fragment usually corresponds to a token or a handful of them; on the wire it is just a couple of lines:

event: content_block_delta
data: {"type":"content_block_delta","index":0,
       "delta":{"type":"text_delta","text":"Fix"}}

When a block ends, a content_block_stop follows, then a message_delta with the stop_reason and the final token counts, and finally message_stop. The typewriter effect you see in the terminal is not an animation that some frontend engineer tuned to feel pleasant. It is the actual cadence of generation reaching you: each delta exists because, tens of milliseconds ago, one more decode iteration completed in a datacenter, and the result was shipped to you essentially as it was produced. On your machine, the loop from the first section now runs in reverse. There, keystrokes became bytes, bytes became JSON, JSON became a request. Here, the CLI reads each event off the socket as it arrives, parses the little JSON payload, appends the fragment to its buffer, and re-renders the reply — markdown parsed live, headings and code blocks colored with the same ANSI escape sequences we met at the very beginning — dozens of times per response, once per repaint, so the text appears to grow under a blinking cursor.

Closing the loop

Very often, though, the stream does not end in prose. Instead of a text block, the model opens a content_block_start of type tool_use, naming a tool — Read, say — and then streams the tool's JSON input in input_json_delta fragments. It is worth pausing on this: the arguments to the tool call are themselves generated token by token, a JSON string assembled through the same forward passes as any sentence, and streamed back with the same event grammar. The stream then closes with stop_reason: "tool_use", and on the server, nothing further happens. The model holds no thread of execution, no pending callback. Whatever happens next is up to the CLI.

What happens next happens entirely on your machine. Claude Code shows you the permission prompt; you press y; the CLI spawns a real process — opens the file, runs the command — against your actual disk. The model never touched it and never could; it only ever emitted text describing what it wanted. The CLI wraps the outcome in a tool_result block, appends it to the conversation, and sends the whole grown context off in a fresh POST. And with that, everything in this article happens again: the TLS handshake (skipped — the connection is still warm), the trip through the internet, the gateway, the batch, the prefill (mostly skipped — the unchanged prefix is a prompt-cache hit), the decode loop. Within one "turn" as you experience it, this round trip commonly repeats several times, sometimes dozens. The whole journey

Let us total up one ordinary exchange — you ask Claude Code to fix a failing test, it reads a file, edits it, and reports back. That is three or four API calls. Across them, on the order of a hundred thousand prompt tokens pass through prefill, the great majority resolved from cache. A few thousand tokens are generated, and each one of those — every word fragment that appears under your cursor — is the product of a complete pass through the model — Anthropic does not publish its size, but frontier models of this class are generally believed to run to hundreds of billions of weights — spread across a team of accelerators working in lockstep. All of it, typically, inside a minute. It is tempting, watching text compose itself in a terminal, to file the whole thing under magic and move on. But nothing in the path we have traced is magic. It is escape codes and JSON, sockets and certificates, routers and load balancers, matrices multiplied very quickly, and a sampler picking one integer at a time — each layer ordinary on its own, each one inspectable if you care to look. The depth of the stack is real, and so is the plainness of every step in it. The next time a reply types itself out in front of you, you will know that the small pause before each word is not a flourish. It is the round trip — the whole round trip — happening again.
