The Way Back

The model has just settled on its next word — but that word is not text yet, only a number, a code for a piece of writing such as Fix. The moment it is chosen it travels back over the same secure connection your request left on, each piece leaving as soon as it is ready — which is why the answer seems to type itself out. Often the reply is not words to read but a request to use a tool: the model asks to open a file or run a command, cannot act itself, and waits while Claude Code asks your permission and does the real work; the result is folded back in and sent again, so one question can loop several times. None of it is magic — only messages and codes, connections and checks, and a great deal of very fast arithmetic. Usually it finishes inside a minute.

The model has committed to a token — an integer in accelerator memory, say id 8543. To become text it runs the journey in reverse: the id is looked up in the vocabulary, mapped back to its bytes (perhaps Fix), and pushed onto the connection the CLI opened with stream: true. The server answers with text/event-stream and trickles the reply out as server-sent events — a message_start, a run of content_block_delta fragments, a message_stop — each shipped as it was produced one decode step ago, which is the typewriter effect. Often the stream is not prose but a tool_use block: the CLI shows a permission prompt, runs a real process against your disk, wraps the outcome in a tool_result, and posts the grown context back — the whole loop again, cheaper on the warm connection and prompt cache. One ordinary “fix the failing test” is three or four calls, usually inside a minute — and none of it is magic, only ordinary parts.

To become text it runs this journey in reverse: the id is looked up in the tokenizer's vocabulary, mapped back to its bytes — perhaps Fix — and pushed onto the HTTPS connection the CLI asked for by setting "stream": true. Instead of one response the server answers with Content-Type: text/event-stream and trickles the reply out as server-sent events, one encrypted chunk at a time, back through the same TLS tunnel.

The stream has a fixed grammar: a message_start carrying the reply's id, model name, and input-token count, then per content block a content_block_start and many content_block_delta events, each a small fragment of text:

event: content_block_delta
data: {"type":"content_block_delta","index":0,
       "delta":{"type":"text_delta","text":"Fix"}}

A content_block_stop closes the block, a message_delta carries the stop_reason and final counts, and message_stop ends it. The typewriter effect is no flourish; it is the cadence of generation itself, each delta shipped as it was produced one decode iteration ago. The sampler ending a forward pass shapes the answer rather than reading it off: the final layer emits a logit vector — one real number per vocabulary entry, on the order of a hundred thousand — which the request's sampling controls massage first. Temperature divides the logits before the softmax: below one sharpens toward the top candidates, above one flattens, toward zero it becomes a deterministic argmax. The softmax makes a probability distribution; nucleus sampling, top-p, keeps the smallest set of tokens whose probabilities sum to p and renormalizes before drawing. Stop sequences are enforced here too — the decoder halts the moment a configured string appears, reporting stop_reason: "stop_sequence".

Only a surviving id is detokenized, which hides a subtlety: the vocabulary is byte-level, so an id maps to raw bytes, not a character, and a multi-byte UTF-8 codepoint — an emoji, say — can straddle two or more tokens. Reading a token's bytes alone would give mojibake at the seams, so the streaming path buffers incomplete sequences, emitting a delta only for complete codepoints and carrying the dangling prefix forward until a later token finishes it.

Those bytes ride the wire as server-sent events ("stream": true): Content-Type: text/event-stream over the kept-alive connection, each frame an event: line and a data: JSON line. The order is strict — message_start with the envelope and input-token usage, per content block a content_block_start, a run of content_block_delta frames and a content_block_stop, then a message_delta bearing the terminal stop_reason and output-token count, and message_stop.

Closing the loop

Often the stream does not end in prose. Instead of text the model opens a content_block_start of type tool_use, naming a tool — Read, say — and streams its JSON input as input_json_delta fragments, generated token by token like any sentence. The stream closes with stop_reason: "tool_use", and on the server nothing further happens: it holds no thread of execution. The rest is up to the CLI: Claude Code shows the permission prompt, you press y, and it spawns a real process against your actual disk — the model never touched it and never could. The CLI wraps the outcome in a tool_result block and posts the grown context in a fresh POST. With that, everything happens again — the handshake skipped on the warm connection, the internet, the gateway, the batch, the prefill mostly skipped on a prompt-cache hit, the decode loop. Within one turn this repeats several times, sometimes dozens. The figure below plays a realistic exchange as a sequence diagram — you, the CLI, and the API — with calls and token counts adding up.

When the model acts rather than speaks, the same grammar carries a different payload. The block that opens is a tool_use block — naming the tool, carrying an id — and its arguments arrive as input_json_delta fragments, partial JSON the client concatenates and parses only once the block closes; they are generated exactly as prose is, token by token through the same forward passes. The turn ends with stop_reason: "tool_use", and here the API's defining property asserts itself: it is stateless, keeping no execution thread and no coroutine waiting on the tool. It has returned a value and forgotten you.

Continuation is the client's job. Claude Code surfaces the permission prompt, you approve, it runs a real process against your filesystem, wraps the output and exit status in a tool_result block, appends both its tool_use turn and that result, and issues a fresh POST with the grown transcript. Everything recurs, but cheaply: the TLS session is still warm, so no new handshake, and the unchanged prefix — system prompt, tool schemas, every prior turn — is a prompt-cache hit, read from cached key/value tensors rather than recomputed, so re-prefill cost falls to the few new tokens the tool result added. That hundred-thousand-token prefill collapsing to near nothing is what makes the multi-round agent loop economical.

The whole journey

Total up one ordinary exchange — fix a failing test: read a file, edit it, report back — and that is three or four API calls carrying on the order of a hundred thousand prompt tokens through prefill, the great majority from cache, against a few thousand generated tokens. Each of those is a complete pass through the model — which Anthropic does not size publicly, though frontier models of this class are believed to run to hundreds of billions of weights — spread across a team of accelerators in lockstep, typically inside a minute. Two latencies govern how this feels. Time to first token is everything before a single delta can leave: the network round trip to the region, time queued for a batch slot, and the prefill pass over the whole prompt to populate the KV cache — compute-bound, scaling with prompt length, the step caching most shortens. Once decoding begins a steadier clock takes over: the per-token cadence of the decode step, memory-bandwidth-bound because every token streams the full weights and growing KV cache through the accelerators once. That is why deltas arrive at an even pace rather than in a burst — the decode loop's period, not a cosmetic animation.
