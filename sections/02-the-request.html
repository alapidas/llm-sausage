The Request

When you press Enter, your message is packed up like a letter and sent to Anthropic’s computers, which do the thinking and reply. Because those computers keep no memory, every letter carries the whole conversation so far, plus a label proving it is from you so only your account is charged. Before the letter goes out, the two computers exchange a quick greeting and agree on a secret code; from then on everything is sealed, so anyone who intercepted it would find only scrambled nonsense — which is why it is safe to send Claude your private notes. The sealed letter is broken into pieces that hop from machine to machine across the internet, none able to read what is inside, until they reach Anthropic’s front door. That door is where we go next.

The CLI has serialized everything the model needs into one JSON document, and it leaves your machine as an ordinary HTTPS POST to https://api.anthropic.com/v1/messages. No private tunnel: a web request like your browser makes all day, only larger. The body carries model, max_tokens (a cap on the reply, not a target), system, the messages array of alternating turns, the tools and their schemas, and stream: true; headers prove who is asking. Before the first byte, DNS resolves the hostname to a nearby edge network, TCP handshakes, and TLS 1.3 establishes shared keys in one round trip, so your API key and source never cross the wire in readable form. The request then travels hop by hop as encrypted packets — each router reads only the address, never the contents — bounded by physics, since light in fiber costs tens of milliseconds across a continent. Next, it reaches Anthropic’s front door.

The document leaves as a single HTTP POST to https://api.anthropic.com/v1/messages over HTTP/2 inside TLS. The JSON is the public Messages API shape; because the endpoint is stateless it is the whole world the model sees — model, a server-enforced max_tokens ceiling, system, messages as alternating turns (tool output returns as a tool_result block in the next user turn), tools with JSON Schema, and stream: true. Prompt caching rides here too: cache_control breakpoints let the server serve a matching prefix from cache, the match a strict prefix over tools, system, messages. First DNS, typically to an anycast edge; then TCP; then TLS 1.3, whose ClientHello carries an ephemeral key_share (usually X25519) so both sides derive the same secret in one round trip, with forward secrecy. Packets then cross your ISP, the backbone, and the edge, every router forwarding on the IP header alone; the payload is opaque ciphertext, but its sizes and timing still leak the shape of the streamed reply, and latency has a hard floor of about 200 km per millisecond in fiber. One field family changes the economics of the agentic loop: prompt caching. Attaching cache_control breakpoints of type ephemeral to content blocks lets the server serve a matching prefix from cache instead of reprocessing it. The match is a strict prefix — the prompt is hashed in the fixed order tools, system, messages, and one differing byte before a breakpoint invalidates everything after — so a well-behaved CLI keeps the stable head byte-identical and appends only at the end. You get up to four breakpoints, and usage reports cache_read_input_tokens and cache_creation_input_tokens to confirm hits; how Anthropic stores those prefixes internally is not public. Before the first byte

A name is not an address, so first comes DNS: your machine asks a resolver what IP api.anthropic.com maps to, and the answer usually belongs not to one datacenter but to an edge network whose points of presence worldwide route the address near you. Then your machine opens a TCP connection — SYN, SYN-ACK, ACK — a round trip that gives both sides agreed sequence numbers, so lost or reordered packets can be repaired later.

In TLS 1.3 your machine sends a ClientHello carrying its half of a key exchange; the server answers with its own half, a certificate proving it is api.anthropic.com, and a Finished message, all in one flight. Your machine verifies the certificate, sends its own Finished, and attaches encrypted application data to that same flight. One round trip, and both ends hold symmetric keys no eavesdropper can derive — which is why your API key and your source code never cross the internet in readable form. A hostname is not a route, so the first work is DNS: your resolver walks the delegation chain and returns A and AAAA records for api.anthropic.com. Large public endpoints are typically fronted by anycast — the same IP announced into BGP from many points of presence at once, the routing fabric delivering your packets to whichever is closest in network terms. Which networks announce it, and where they terminate the connection, is not public for Anthropic specifically; this is how large API services are commonly fronted. TCP then handshakes — SYN, SYN-ACK, ACK — establishing sequence numbers so loss and reordering can be repaired.

TLS 1.3 then secures the channel in one round trip. The ClientHello carries a key_share — your half of an ephemeral Diffie-Hellman exchange, typically over X25519 — and the server replies with its own, from which both sides derive the same secret, followed by its certificate chain and a Finished, now encrypted. Your machine validates that chain to a trusted root, confirms the certificate's subject-alternative-name list covers the host, and appends encrypted application data to the same flight. The ephemeral keys give forward secrecy; a resumed session reuses a pre-shared-key ticket to collapse the handshake further. One detail worth naming: the Server Name Indication — the cleartext api.anthropic.com in the ClientHello — stays visible unless Encrypted Client Hello is used.

The sealed letter does not travel in one piece: it is broken up, and each piece is passed from machine to machine — your home router, your provider's equipment, a chain of big routers forming the internet's backbone. Every machine reads the address outside; none can read what is sealed inside.

None can beat distance: the pieces move at nearly the speed of light, yet crossing an ocean takes noticeably longer than crossing a city. That toggle matters: an agentic session is a loop — the model asks for a tool, the CLI runs it and sends the result back — so HTTP keep-alive holds the connection open, and every request after the first skips DNS and both handshakes.

Hop by hop

Split into packets, the request travels a path you never see, from your Wi-Fi router to the edge point of presence your DNS answer named. Each router reads only the IP header — destination, a hop counter, a few flags — and forwards the packet on; the payload is TLS ciphertext, so a router could copy every byte and learn nothing beyond who is talking to whom.

This has a hard floor: light in fiber travels about 200 km per millisecond, so crossing a continent costs tens of milliseconds — which is why edge networks terminate close to you. Below, watch a packet make the trip, hover over or tap any node to see what it can read, and drag the slider from same-city to trans-oceanic distances to feel geography dominate the total.

Over that connection the CLI speaks HTTP/2: the request becomes binary frames, headers are compressed with HPACK, and the whole agentic loop — request, tool call, result, next request — is multiplexed over the single kept-alive connection, so every turn after the first pays only one-way travel rather than another setup. Each frame is split into IP packets that cross your ISP's access and aggregation layers, the backbone, and the edge terminating your session. Every router forwards on the destination address in the IP header alone, decrementing the TTL, with the route between autonomous systems chosen by BGP over peering and transit links.

To all of them the payload is opaque ciphertext, which hides the contents — your key, your code, the model's tokens — but not the metadata. Because the response streams token by token, the sizes and timing of the ciphertext frames trace the shape of the generation, so an observer counting bytes and clocking arrivals learns roughly how long the answer is and how fast it comes; padding and framing blunt this without erasing it. Latency, meanwhile, has a floor set by physics: light in fiber moves about 200 km per millisecond, so a transcontinental leg costs tens of milliseconds — the reason terminating your connection near you is worth the trouble.
