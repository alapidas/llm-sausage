The Forward Pass

Your prompt is now a stack of vectors — one per token, each a list of a few thousand numbers. What receives them is surprisingly repetitive. A transformer is a tall pile of identical layers, some dozens of them, and every layer holds the same two sublayers. The first is attention, the only place where tokens exchange information with one another. The second is a feed-forward network — an MLP, in fact the bulkier of the two — which transforms each token's vector on its own, ignoring its neighbors. That is essentially the whole architecture, applied over and over.

How the layers connect matters as much as what they compute. Each token's vector flows upward along the residual stream. A sublayer does not replace the vector; it reads it, computes a small correction, and adds that correction back in. The vector leaving the top is the original embedding plus dozens of accumulated refinements — a running draft, never rewritten from scratch. Every token makes this trip at once, all positions in parallel — a large part of why GPUs suit this work.

Attention

Attention is where the reading happens, and the mechanism is simpler than its reputation. From its current vector, each token computes three new vectors against three learned matrices: a query — "here is what I am looking for" — a key — "here is what I contain" — and a value — "here is what I will hand over if asked." A token's query is compared against the key of every token up to and including itself, one dot product per pair — a single number measuring how well the two vectors line up. Those scores pass through a softmax into positive weights summing to one, and the token pulls in a mixture of the others' values in those proportions — heavily where keys matched its query, faintly elsewhere. That mixture is the correction added to its residual stream.

Two details complete the picture. First, causality: a token may attend only to itself and to positions before it, never forward — the future is masked out. Second, each layer runs many of these operations side by side, in separate heads, each with its own query, key, and value matrices. One distribution

At the top of the stack, one more multiplication. The final vector of the last token is projected against the unembedding — a matrix with one row per vocabulary entry — producing one score per token: the logits. A last softmax converts them into probabilities. A machine with billions of parameters has just run end to end across your whole context, and all that work has produced exactly one thing: a probability distribution over the next token. Not an answer — one distribution.

Something must turn that distribution into a choice. Always taking the most likely token — greedy decoding — produces stilted, repetitive text that falls into loops. Instead, the next token is sampled from the distribution, after reshaping it with a temperature: the logits are divided by T before the softmax. As T approaches zero the distribution sharpens toward its favorite; as it grows it flattens toward noise. A companion trick, top-p, discards the long tail of barely-plausible tokens before sampling. One token is chosen and appended — then the whole forward pass runs again on the longer context for the next distribution. Every visible word of a reply is produced this way, one token at a time.

The KV cache

"Runs again" should worry you. To produce token N + 1, attention needs the keys and values of all N earlier tokens — and naively, every step would recompute them all, though they depend only on the unchanged prefix. So they are computed once and kept: every token's keys and values, for every layer and head, held in accelerator memory as the KV cache. Each step computes a query, key, and value for the newest token alone, attends over the cached rest, and appends its own K and V.

This one optimization shapes the economics you feel as a user. The cache is large — a few hundred kilobytes per token for a big model, so a long agent session runs to gigabytes — and on every step it must be streamed out of memory, along with the model's weights, which are far larger still. Moving all those bytes, not the arithmetic, is what limits generation speed, and the growing cache is why long contexts respond more slowly and cost more. It is also, in essence, what prompt caching — which we met in the datacenter section — preserves between calls of the agent loop: the prefix's computed state stays warm so the next request need not rebuild it. Production models refine this picture in ways we are skipping: grouped-query attention shares keys and values across heads to shrink the cache, mixture-of-experts layers route each token to a subset of the MLP weights, and speculative decoding drafts several tokens cheaply and verifies them in one pass. We are also skipping how the model knows token order: position information is mixed into the queries and keys — in modern models by rotating them as a function of position, a scheme called RoPE — which is what lets a head prefer "the previous token" at all. None of it changes the shape of the story here.

However the details fall, at the end of the pass there now exists a token id that did not exist a moment ago — and it has to make its way back out of the datacenter to your terminal.
