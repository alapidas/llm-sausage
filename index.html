A program drawing a picture of a prompt

The interface greeting you is not the terminal’s own prompt but a JavaScript program drawing a text-mode interface. Normally a terminal runs in what Unix calls canonical mode: the operating system buffers your keystrokes into a line, handles backspace, echoes each character, and hands over the finished line only at Enter. An interactive interface cannot work that way, so Claude Code switches into raw mode, where every key reaches the CLI process immediately as a raw byte or two, nothing echoed automatically.

Every character you see is thus the program rendering, not the terminal echoing. When a byte arrives, the CLI updates its state and repaints the affected region by writing ANSI escape sequences to standard output. All of it is redrawn many times a second, on the same channel your shell uses to print ls. The window is not the terminal’s prompt but a program that has seized control of it (npm-distributed for Node.js, or shipped with its own runtime). A Unix terminal’s driver defaults to canonical mode, governed by a termios struct whose flags set the line discipline: ICANON buffers bytes until a newline before waking the reader, and ECHO prints each byte back. A text UI cannot wait for newlines, so the CLI switches the descriptor to raw mode: it clears ICANON and ECHO, disables signal generation, and sets VMIN = 1, VTIME = 0 so read() returns on the first available byte, not a whole line. Keystrokes then arrive at once on standard input, echoed only if the program echoes them.

Most keys arrive as the obvious byte; the interesting ones do not. The up arrow sends three bytes — ESC, [, A, that is \x1b[A — a CSI (Control Sequence Introducer) escape sequence; Home, End, the function keys, and modified chords each encode their own, which the program parses to recover the keypress. Output uses the same alphabet in reverse — \x1b[H homes the cursor, \x1b[2K erases a line, \x1b[38;2;…m sets a truecolor foreground — on the very channel your shell uses to print ls.

Built on a React renderer for the terminal (Ink), the program keeps a virtual screen and re-renders it like frames: each state change diffs against the last and emits only the escape sequences to reconcile them. On resize the kernel delivers SIGWINCH; the process re-reads rows and columns through the TIOCGWINSZ ioctl and repaints. An agent, not an oracle

Where do those tool results come from? The model cannot open your files or run your tests; it only receives and produces text. What makes Claude Code an agent is the program wrapped around it. When the model wants auth.spec.ts, it replies not with an answer but with a tool_use request. The CLI checks it against your permission settings, prompting you when required, runs the tool locally, packages the output into a tool_result message, and sends the whole context back. The Request

When you press Enter, your message is packed up like a letter and sent to Anthropic’s computers, which do the thinking and reply. Because those computers keep no memory, every letter carries the whole conversation so far, plus a label proving it is from you so only your account is charged. Before the letter goes out, the two computers exchange a quick greeting and agree on a secret code; from then on everything is sealed, so anyone who intercepted it would find only scrambled nonsense — which is why it is safe to send Claude your private notes. The sealed letter is broken into pieces that hop from machine to machine across the internet, none able to read what is inside, until they reach Anthropic’s front door. That door is where we go next.

The CLI has serialized everything the model needs into one JSON document, and it leaves your machine as an ordinary HTTPS POST to https://api.anthropic.com/v1/messages. No private tunnel: a web request like your browser makes all day, only larger. The body carries model, max_tokens (a cap on the reply, not a target), system, the messages array of alternating turns, the tools and their schemas, and stream: true; headers prove who is asking. Before the first byte, DNS resolves the hostname to a nearby edge network, TCP handshakes, and TLS 1.3 establishes shared keys in one round trip, so your API key and source never cross the wire in readable form. The request then travels hop by hop as encrypted packets — each router reads only the address, never the contents — bounded by physics, since light in fiber costs tens of milliseconds across a continent. The document leaves as a single HTTP POST to https://api.anthropic.com/v1/messages over HTTP/2 inside TLS. The JSON is the public Messages API shape; because the endpoint is stateless it is the whole world the model sees — model, a server-enforced max_tokens ceiling, system, messages as alternating turns (tool output returns as a tool_result block in the next user turn), tools with JSON Schema, and stream: true. Prompt caching rides here too: cache_control breakpoints let the server serve a matching prefix from cache, the match a strict prefix over tools, system, messages. First DNS, typically to an anycast edge; then TCP; then TLS 1.3, whose ClientHello carries an ephemeral key_share (usually X25519) so both sides derive the same secret in one round trip, with forward secrecy. Packets then cross your ISP, the backbone, and the edge, every router forwarding on the IP header alone; the payload is opaque ciphertext, but its sizes and timing still leak the shape of the streamed reply, and latency has a hard floor of about 200 km per millisecond in fiber. Before the first byte

A name is not an address, so first comes DNS: your machine asks a resolver what IP api.anthropic.com maps to, and the answer usually belongs not to one datacenter but to an edge network whose points of presence worldwide route the address near you. Then your machine opens a TCP connection — SYN, SYN-ACK, ACK — a round trip that gives both sides agreed sequence numbers, so lost or reordered packets can be repaired later.

In TLS 1.3 your machine sends a ClientHello carrying its half of a key exchange; the server answers with its own half, a certificate proving it is api.anthropic.com, and a Finished message, all in one flight. Your machine verifies the certificate, sends its own Finished, and attaches encrypted application data to that same flight. One round trip, and both ends hold symmetric keys no eavesdropper can derive — which is why your API key and your source code never cross the internet in readable form. A hostname is not a route, so the first work is DNS: your resolver walks the delegation chain and returns A and AAAA records for api.anthropic.com. Large public endpoints are typically fronted by anycast — the same IP announced into BGP from many points of presence at once, the routing fabric delivering your packets to whichever is closest in network terms. Which networks announce it, and where they terminate the connection, is not public for Anthropic specifically; this is how large API services are commonly fronted. TCP then handshakes — SYN, SYN-ACK, ACK — establishing sequence numbers so loss and reordering can be repaired.

TLS 1.3 then secures the channel in one round trip. The ClientHello carries a key_share — your half of an ephemeral Diffie-Hellman exchange, typically over X25519 — and the server replies with its own, from which both sides derive the same secret, followed by its certificate chain and a Finished, now encrypted. Your machine validates that chain to a trusted root, confirms the certificate's subject-alternative-name list covers the host, and appends encrypted application data to the same flight. The ephemeral keys give forward secrecy; a resumed session reuses a pre-shared-key ticket to collapse the handshake further. One detail worth naming: the Server Name Indication — the cleartext api.anthropic.com in the ClientHello — stays visible unless Encrypted Client Hello is used.

Hop by hop

Split into packets, the request travels a path you never see, from your Wi-Fi router to the edge point of presence your DNS answer named. Each router reads only the IP header — destination, a hop counter, a few flags — and forwards the packet on; the payload is TLS ciphertext, so a router could copy every byte and learn nothing beyond who is talking to whom.

A frontier model's weights run to hundreds of gigabytes, far beyond the tens of gigabytes of high-bandwidth memory on one accelerator, so the model is sharded across chips that act as one logical server, cut along two axes. Tensor parallelism splits individual weight matrices: each chip holds a vertical slice of every layer and computes its partial for each token, and the partials are combined by an all-reduce before the next layer. Pipeline parallelism instead splits by depth: each chip holds a block of consecutive layers and tokens flow down the pipeline. Its cross-chip traffic is far smaller — only activations at block boundaries — so it tolerates slower links and spans nodes, at the cost of a "bubble" of idle chips that in-flight micro-batches must hide. Where the vocabulary comes from

Nobody chooses those fragments by hand; they are learned by byte-pair encoding, a procedure that is almost embarrassingly simple. Start with the 256 possible bytes, so every text is a sequence of single-byte tokens, then scan a huge corpus for the most frequent adjacent pair — in English, roughly t then h — fuse it into a new token, and rewrite the corpus. The vocabulary is the output of byte-pair encoding. Seed it with the 256 byte values, express the corpus as single-byte tokens, then repeatedly find the most frequent adjacent pair and add it as a new token, recording the order each was learned. That order is the merge rank, essentially the whole tokenizer at inference time: to encode new text you apply merges greedily, always taking the lowest-ranked applicable merge first and replaying until no ranked pair remains. Encoding is therefore deterministic and roughly linear in length, and whatever was common in training stays cheap forever after.

Two consequences follow. First, the 256-byte floor never leaves the vocabulary, so byte fallback is always available: an unseen emoji, a rare script, or arbitrary binary junk decomposes into its UTF-8 bytes rather than failing — there is no unknown-token case, only inputs that cost more tokens. Second, vocabulary size is a genuine tradeoff: a larger vocabulary packs more text into fewer tokens, shortening sequences and saving compute in the length-bound layers, but it inflates the embedding and output-projection matrices and spends parameters on rare merges seen little in training. Attention

Attention is simpler than its reputation. Each token forms three vectors against learned matrices — a query ("what I seek"), a key ("what I hold"), a value ("what I give") — and its query is dotted against the key of every token up to itself. A softmax turns those scores into weights summing to one, and the token pulls in a mixture of the others' values in those proportions, the correction added to its residual stream.

Two details complete it: a token may attend only to itself and earlier positions, never the future, and each layer runs many such operations in parallel heads. In the demonstration below, hover or tap any word to see where it attends under three hand-made heads.

Attention, the only sublayer that mixes information across positions, is built from three learned projections of the normalized stream: Q = xW_Q, K = xW_K, V = xW_V. The output is the scaled dot-product form softmax(QKᵀ / √d) V, with d the head dimension: each query dots against every key, the √d divisor keeps the scores from saturating the softmax, and the weights average the values. Causal masking sets future positions' pre-softmax scores to −∞, so position i attends only to positions ≤ i and can be trained without seeing its future.

This runs as multi-head attention: the projections split into independent heads, each in its own d-dimensional subspace, so heads specialize in different relations. Grouped-query attention, now common, lets many query heads share one key/value head — barely denting quality while shrinking the key/value footprint several-fold, key to the cache ahead. Order enters here too, not as a position vector at the input but as rotary embeddings (RoPE): each query and key is rotated by an angle proportional to position, so their dot product depends only on relative offset. The KV cache

"Runs again" should worry you. To produce token N + 1, attention needs the keys and values of all N earlier tokens, which depend only on the unchanged prefix — so they are computed once and kept in memory as the KV cache. Each step computes a query, key, and value for the newest token alone, attends over the cached rest, and appends its own.

This shapes the economics you feel. Either way, the pass has produced a token id that did not exist a moment ago — and it must make its way back to your terminal.

At the top the residual stream is normalized once more and multiplied by the unembedding matrix, of shape [d_model, vocab] — often the input embedding transposed and reused. For generation only the last position matters: its final vector becomes the logits, one score per vocabulary entry, the entire output of the forward pass. Turning it into text is a separate, cheap step: dividing the logits by a temperature T before the softmax rescales the distribution — T → 0 collapses toward the argmax, larger flattens toward uniform — while top-p truncates to the smallest set of tokens whose probability sums to p. Greedy decoding (the T → 0 limit) tends to loop, so a modest temperature with top-p is the usual compromise.

Producing each token re-runs the whole stack, but the prefix’s keys and values never change, so they are computed once and retained — the KV cache — about 2 × layers × kv_heads × head_dim × bytes per token. Take 80 layers, 8 key/value heads after grouped-query attention, head dimension 128 in 2-byte bf16: roughly 320 KB per token, so a 100,000-token context holds about 32 GB of cache beside the weights. The consequence is general: generation is bound by memory bandwidth, every step streaming the weights and the whole cache out of high-bandwidth memory once, and moving those bytes sets the pace — why long contexts cost more and run slower.

Production models fold in refinements skipped here — grouped-query attention shares keys and values across heads, mixture-of-experts layers route each token to a subset of the MLP weights, and speculative decoding drafts several tokens and verifies them in one pass. The specifics — layer and head counts, RoPE — are typical of modern transformers, not claims about any production model, whose internals are not public. None of it changes the shape of the story.
