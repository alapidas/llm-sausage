How the Sausage Gets Made

What actually happens when you talk to Claude Code — from the keystroke in your terminal, across the wire, into a datacenter, and through a neural network, one token at a time.

Your Terminal

It starts with a sentence. You open a terminal, run claude, and ask for something ordinary — fix the failing test, rename this function. You press Enter, a spinner turns for a moment, and an answer streams back character by character, as if someone on the other end were typing. It feels like a chat window. It is nothing like a chat window. Between your keystroke and the first character of the reply, your sentence will be folded into a document hundreds of thousands of characters long, serialized into JSON, carried across the internet to a datacenter, sliced into tokens, pushed through layer after layer of matrix arithmetic on GPUs, and turned back into text one small piece at a time. In this article we will follow that one request the whole way there and back. We begin on your own machine, where — before any AI is involved at all — a surprising amount is already going on.

A program drawing a picture of a prompt

The interface that greets you is not the terminal’s own prompt. It is a Node.js process that has taken over the window and is drawing a text-mode user interface, somewhat the way a game draws frames. Ordinarily a terminal runs in what Unix calls canonical mode: the operating system collects your keystrokes into a line buffer, handles backspace for you, echoes each character to the screen, and only hands the finished line to the program when you press Enter. An interactive interface cannot work that way — it wants to react to every keystroke, offer completions, move a selection with the arrow keys. So Claude Code switches the terminal into raw mode. From then on every key you press is delivered to the Node process immediately, as one or a few raw bytes on standard input, and nothing is echoed automatically. If the program did nothing, your typing would be invisible.

Every character you see is therefore the program rendering, not the terminal echoing. When a byte arrives, the CLI updates its internal state — the text of your draft message, the cursor position — and repaints the affected part of the screen by writing ANSI escape sequences to standard output: move the cursor here, switch to this color, erase to the end of the line, write these characters. The blinking block cursor, the border around the input box, the spinner that appears while a request is in flight — all of it is redrawn, many times a second, through the same output channel your shell uses to print the results of ls. In the demonstration below, a request types itself into a miniature terminal, and the lane underneath tracks each keystroke: the key event, the raw byte handed to the Node process, and a running count of screen redraws. Watch the counter keep climbing after Enter — even the idle spinner is repaint after repaint.

What Enter actually sends

When you press Enter, it would be reasonable to assume the CLI sends your sentence to the model. It does — but your sentence rides along at the very end of something much larger. The API is stateless: between calls, the model retains no memory of your project or your earlier messages. So on every single request, the CLI assembles the full context from scratch. First comes a large system prompt: several thousand words of instructions about how to behave — how to format answers, when to ask before acting, how to use tools. Then the tool definitions: every tool the model may request — Bash, Read, Edit, Grep, and more — each described as a JSON schema, its parameters and their types spelled out in full. Then project memory: files such as CLAUDE.md, notes about how this codebase works. Then the entire conversation so far: every message, every reply, and every tool result — file contents, command output, search matches — accumulated over the session. And then, at last, your new message.

The figure below lays this assembly out as a single horizontal bar: the model’s context window, with room for 200,000 tokens. A token is a short chunk of text, roughly three-quarters of an English word; we will meet tokens properly in a later section. Drag the slider to extend the conversation, and hover over or touch each segment to see what it holds. Notice how quickly the history grows — each turn adds not only what was said but everything the tools returned — and how comically thin the green sliver of your new message remains throughout.

The segment sizes here are illustrative rather than measured; they change between versions. But the system prompt and tool schemas together typically account for tens of thousands of tokens before you have typed anything at all.

An agent, not an oracle

There is a second surprise hiding in that bar: the tool results. Where do they come from? The model itself cannot open your files, run your tests, or execute anything at all. It receives text and produces text — that is the entirety of its powers. What makes Claude Code an agent is the program wrapped around the model. When the model wants to look at auth.spec.ts, it replies not with an answer but with a structured tool_use request — in effect, “please run Read on this path.” The CLI checks that request against your permission settings (asking you first when the rules require it), executes the tool locally on your machine, packages the output into a tool_result message, appends it to the conversation, and sends the whole context to the API again. The model, now seeing the file’s contents, may ask to run the test suite — and the cycle repeats.

In the animation below you can watch this loop turn. A single “fix the failing test” routinely fans out into half a dozen API calls before the model finally responds with plain prose instead of another tool request. How a finished reply becomes an edit on your disk is a story for the final section; for now it is enough to see the shape of the cycle.

Step back and notice what this architecture implies. Everything the model will ever know about your request is inside that one growing document. It cannot peek at your screen, browse your filesystem in the background, or recall what you asked yesterday. If a fact is not in the context — placed there by the system prompt, a memory file, or a tool result the model itself asked for — then for the model it does not exist. There is no hidden channel. The context is the interface.

At this point the CLI serializes the assembled context into the body of a JSON request addressed to Anthropic’s API. In the next section, we follow it out of your machine.

The Request

By the end of the previous section the CLI had gathered everything the model needs to see and serialized it into a single JSON document. Now that document has to leave your machine. It leaves as one ordinary HTTPS POST to https://api.anthropic.com/v1/messages — the same endpoint every other program that talks to the Claude API uses. There is no private tunnel and no special protocol; it is a web request like the ones your browser makes all day, only with an unusually large body and, as we will see, an unusually long-lived response.

It is worth dwelling on what the body contains, because the API is stateless: the server keeps no memory of your conversation between calls. Everything the model will know arrives inside this one request. The model field names which model should answer. max_tokens caps how long the reply may grow — a stop condition, not a target. system carries the CLI's standing instructions. messages is the conversation itself, an array that strictly alternates between user and assistant turns; when the model previously asked to run a tool, the tool's output travels back embedded in the next user turn as a tool_result content block, so from the API's point of view even the output of a shell command is just something you said. tools lists every tool the model may request, each described by a JSON Schema so that legal arguments are precisely defined. And stream is set to true — we will come back to that. Alongside the body ride a few headers: x-api-key — or an OAuth bearer token, if you signed in with a subscription — proves who is asking, anthropic-version pins the API's behavior to a known date, and content-type declares the body to be JSON.

In the figure below you can hover over or tap each highlighted part of an abbreviated request to see what it is for. A real body from a working session would run to hundreds of kilobytes, so the long strings are elided here.

Before the first byte

Sending a request to api.anthropic.com sounds like a single action, but a name is not an address. First comes DNS: your machine asks a resolver what IP address the name maps to, and the answer is usually not a computer in one particular datacenter but an address belonging to an edge network — infrastructure with points of presence spread around the world, arranged so that the address you get routes to somewhere close to you. With an address in hand, your machine opens a TCP connection: it sends a SYN, the far end answers SYN-ACK, and your machine replies ACK. That single round trip gives both sides agreed-upon sequence numbers, so lost or reordered packets can be detected and repaired later.

Only now does security enter. In TLS 1.3 your machine sends a ClientHello that already includes its half of a key exchange; the server answers with its own half, a certificate proving it really is api.anthropic.com, and a Finished message — all in one flight. Your machine verifies the certificate, sends its own Finished, and can attach encrypted application data to that very same flight. One round trip, and both ends hold symmetric keys that no eavesdropper can derive. This is the reason your API key, your source code, and the model's reply never cross the internet in readable form. The demonstration below replays the whole exchange as a sequence diagram, with a clock accumulating a plausible 30 ms per round trip. Try the toggle: when the CLI reuses a connection it already has open, everything above the final arrow simply disappears.

That toggle matters more than it may seem. An agentic session is not one request; it is a loop — model asks for a tool, CLI runs it, sends the result back, and so on, sometimes dozens of times. The CLI therefore keeps the connection warm with HTTP keep-alive, and every request after the first skips DNS, the TCP handshake, and the TLS handshake entirely, paying only the one-way travel time.

Hop by hop

The request itself, split into packets, now travels a path you never see: from your laptop to your Wi-Fi router, through your ISP's equipment, across a handful of backbone routers, and into the edge point of presence that your DNS answer pointed at. Each router along the way makes its decision by reading only the IP header — the destination address, a hop counter, a few flags — and forwards the packet on. The payload is TLS ciphertext to every one of these middleboxes; they could copy every byte and learn nothing beyond who is talking to whom, and roughly how much.

The time this takes has a hard floor: light in optical fiber travels at roughly two thirds of its speed in vacuum, about 200 km per millisecond. Crossing a continent therefore costs tens of milliseconds no matter how clever the routing, which is exactly why edge networks try to terminate your connection close to you. In the figure below you can watch a packet make the trip, hover over or tap any node to see what it can read, and drag the slider from same-city to trans-oceanic distances to feel how geography dominates the total.

We drew five hops; a real traceroute typically shows ten to twenty, and the exact machinery fronting api.anthropic.com — which networks announce it, where TLS is terminated — is not public. What we describe here is how large API services are typically fronted; the physics of the trip is the same either way.

One last detail before we let the packets go. Because the request said stream: true, the response will not be a single document that arrives when the model is done. It will be a long-lived HTTP response carrying server-sent events, and the connection we worked so hard to establish stays open for the entire generation, delivering the reply piece by piece as it is produced. We will unpack that stream in the final section.

For now the packets have arrived at Anthropic's front door — and a large-scale inference service has to decide, in a few milliseconds, what to do with them.

Into the Datacenter

We left your request as a stream of encrypted packets, addressed to api.anthropic.com, disappearing into the network. Now they arrive. The first machines to touch them are edge servers whose job is mundane but essential: they hold the private keys for the TLS session, so this is where the encryption is finally peeled away and your request becomes readable JSON again — inside Anthropic's infrastructure and nowhere else along the way.

The decrypted request then meets an API gateway. Before any model sees a single byte, the gateway authenticates the API key in the x-api-key header, checks it against rate limits and usage quotas, and validates the JSON body — is the model name real, is max_tokens sane, does the message list parse? Requests that fail any of these checks are turned away right here, with the 401 or 429 answered in a few milliseconds without waking a single accelerator. Requests that pass are handed to a load balancer, which picks one backend out of a large fleet of model servers — and because each fleet holds one model's weights, a request for Opus and a request for Haiku part ways at this point and travel to different pools of machines.

One honest caveat before we go further: the exact layout of Anthropic's production stack is not public, and we won't pretend to know it. Everything in this section describes how large-scale LLM inference services are generally built — the standard, well-documented machinery of the field. The real system surely differs in details, but these are the mechanisms any service at this scale relies on.

In the demonstration below, you can watch this front door at work. Requests stream in from the left as colored dots, pause briefly at the gateway while their credentials are checked — the occasional red one is bounced for a bad key or an exceeded rate limit — and the survivors are routed to whichever server has capacity to spare. Your own request is the orange dot. Drag the slider to raise the incoming rate and watch the load bars respond.

A model too big for one chip

Why does the request need a special "model server" at all, rather than any machine in the building? Because a frontier model is mostly one enormous block of numbers: its weights run to hundreds of gigabytes, far more than any single GPU or accelerator can hold. So the model is sharded — sliced across a group of accelerators wired together with very fast interconnects, and that whole group acts as one logical server. When we drew a single box labeled "model server" above, the box really contains several chips, each holding its slice of the weights, and every single token generated for you involves all of them working in concert.

Those weights were loaded once, when the server started, and they stay resident in accelerator memory permanently. This inverts the usual picture of a web service, where data is fetched to wherever the code runs. Here the model is far too heavy to move, so it never does: your few kilobytes of request travel to the weights, flow through them, and leave. The weights are the factory; your request is the workpiece.

Sharing the assembly line

Here is the fact that shapes everything about how these servers are run: an accelerator performs the arithmetic for one generation step so quickly that serving a single user would leave it idle almost all the time. The expensive part of each step is streaming those hundreds of gigabytes of weights through the compute units — and having streamed them, the chip can apply them to one request or to dozens for nearly the same cost. So serving systems batch: many strangers' requests ride through the very same forward passes, sharing each read of the weights.

The refinement that makes this practical is called continuous batching. Requests don't wait for a full busload to assemble; they join and leave the batch at the granularity of a single token. The moment one request finishes and frees its slot, another slides in mid-stream, and the assembly line never pauses. The figure below shows one server's batch as a timeline: each row is a slot, each column one forward pass, and each rounded bar one request — a short bright chunk when its prompt is absorbed, then one small cell per generated token. Yours is again in orange. Raise the arrival rate and watch the occupancy and the combined token rate climb.

Notice what the readout implies: the server's total output grows almost linearly with occupancy, while each individual request still gets a token on every pass. This is why batching is universal — it multiplies the useful work per pass without slowing any one user down much. You have almost certainly never noticed the strangers sharing your forward passes, and that is the point.

Prefill and decode

Within each request, the work splits into two very different phases, and the split explains something you have felt every time you use Claude. First comes prefill: the entire prompt — every token of it — is pushed through the model in one massively parallel pass, because all the prompt's tokens are already known and can be processed at once. This phase is limited by raw arithmetic, and its duration is your time to first token. Then comes decode: tokens are generated one at a time, each requiring its own complete pass through every layer, each pass limited not by arithmetic but by how fast the weights can be streamed from memory. That per-pass time sets your tokens-per-second. One big gulp, then many small sips — that is the pause-then-stream rhythm you know. In the demonstration below, you can replay both phases in slow motion, and lengthen the prompt to see the first-token wait grow while the decode rate stays flat.

Prefill is also where a well-known optimization lives. If the beginning of your prompt is identical to one the server processed moments ago — and in an agent loop it almost always is, since every iteration resends the same system prompt, tool definitions, and conversation so far — the service can keep the computed internal state of that prefix and skip re-prefilling it. This is prompt caching, and it is not speculation: it is a documented feature of Anthropic's API, which is exactly why Claude Code marks the stable prefixes of its requests as cacheable. It is the difference between an agent loop that re-reads the whole transcript from scratch on every turn and one that only pays for what's new.

There is one thing we have quietly glossed over. We keep saying the model processes "tokens," but your request is still text — bytes of UTF-8. Before the first layer of the model can touch it, that text has to become numbers. How that happens is our next stop.

Tokens

The request has cleared the routing layers and reached a machine that holds the model's weights. What it carries is still just text — your prompt, the conversation so far, the tool definitions, all of it one long string of bytes. A neural network cannot do anything with a string of bytes directly. It computes with numbers, so before anything else happens the text is translated: it is cut into tokens, and each token is exchanged for an integer.

A token is not a character, and it is not quite a word. It is an entry in a fixed vocabulary of text fragments — modern models use vocabularies on the order of a hundred thousand entries or more. Common words are single tokens. So are common prefixes and endings, frequent letter pairs, punctuation runs, and — perhaps surprisingly — words with a leading space attached: the and ␣the are two different entries, because inside running text a word almost always arrives with the space that precedes it. Rare words don't get an entry of their own; they are assembled from several smaller pieces. In the demonstration below you can type any text and watch it split. Each colored pill is one token, with its integer id underneath, and the ␣ marks show where a space belongs to the token itself:

A few things are worth noticing as you play with it. The code preset tokenizes remarkably economically: function, return, a run of indentation, the (); at the end of a call — each is a single token, because tokenizers are trained on enormous amounts of source code and these sequences occur constantly. A long rare word shatters into many small fragments. Numbers shatter too, often digit by digit, which is one honest reason arithmetic is not a model's most natural skill. And the emoji becomes raw bytes: the vocabulary always keeps the 256 individual byte values as a floor, so any input whatsoever — any language, any binary junk — can be represented. There is no such thing as an unknown word, only a word that costs more tokens.

Where the vocabulary comes from

Nobody sits down and chooses those hundred thousand fragments by hand. They are learned, with a procedure called byte-pair encoding that is almost embarrassingly simple. Start with a vocabulary of just the 256 possible bytes, so every text is a long sequence of single-byte tokens. Now scan a huge corpus of text and find the pair of adjacent tokens that occurs most often — in English that is something like t followed by h. Fuse that pair into a new token, add it to the vocabulary, and rewrite the corpus using it. Then repeat: find the most frequent adjacent pair in the rewritten corpus, fuse it, repeat again, on the order of a hundred thousand times. Frequent sequences condense into single tokens early; rare ones never do. In the demonstration below you can watch this play out on a tiny corpus, one merge at a time — notice how quickly th, the, and then ␣the with its leading space crystallize out of the character soup:

This training history is exactly why tokenization has the texture you saw in the first demonstration. The merges are ranked by frequency, and encoding new text just replays them in order, so whatever was common in the training corpus is cheap forever after. It also explains the sharp edges. Hello and ␣hello are unrelated entries, so capitalization and spacing genuinely change what the model receives. Your API bill and the context window are both measured in tokens, so the same program is cheaper to send than the same number of characters of, say, transliterated Klingon. And when you ask a model how many letters are in a word, you are asking about characters it never actually saw — it received one or two opaque ids, and any spelling knowledge it has was absorbed indirectly during training rather than read off the input.

From ids to vectors

So the tokenizer hands the model a list of integers. An integer, though, is a barren thing: id 4,072 is not "more similar" to id 4,073 than to id 90,000. The very first thing the model proper does is exchange each id for something far richer. It holds a giant table — the embedding matrix — with one row per vocabulary entry, and each row is a vector of thousands of numbers. Looking up a token is nothing more than reading row id of that table. Everything the model can be said to know about a token in isolation lives in that vector, and the knowledge is encoded geometrically: tokens with related meanings sit near each other, and consistent directions in the space capture consistent relationships. Nobody designed this arrangement. During training the model is endlessly graded on predicting the next token, and nudging related tokens toward similar vectors happens to make those predictions better, so the geometry simply emerges. In the demonstration below you can tap the words on the map to select a token and inspect a schematic slice of its vector — notice how the neighborhoods organize themselves, and how nearby words have similar-looking columns of values:

Anthropic does not publish the production tokenizer used by its models, so the tokenizations in this section come from a small illustrative byte-pair tokenizer with a few hundred merges built for these figures. The behavior it exhibits — greedy merges, whitespace sensitivity, rare words shattering, bytes as a fallback — is faithful to how real tokenizers work, but the exact token boundaries and ids will differ from what the real vocabulary produces.

Your entire request is now a sequence of vectors, thousands of numbers per token, lined up in order. That sequence is the actual input to the transformer, and pushing it through — the forward pass — is where we go next.

The Forward Pass

Your prompt is now a stack of vectors — one per token, each a list of a few thousand numbers. What receives them is surprisingly repetitive in shape. A transformer is a tall pile of identical layers, typically some dozens of them, and every layer contains the same two sublayers. The first is attention, the only place where tokens exchange information with one another. The second is a small feed-forward network — an MLP — which transforms each token's vector on its own, paying no attention to its neighbors. That is essentially the entire architecture, applied over and over.

The way the layers connect matters as much as what they compute. Each token's vector flows upward through the stack along what is called the residual stream. A sublayer does not replace the vector; it reads it, computes a small correction, and adds that correction back in. The vector that leaves the top of the stack is the original embedding plus dozens of accumulated refinements — a running draft that gets annotated on every floor but never rewritten from scratch. In the figure below you can watch three tokens ride their residual streams through a small eight-layer stack; the slider chooses which token to follow, and as its vector passes each sublayer you can see the side branch that reads it and merges a contribution back in with a small addition:

Every token in your context makes this trip at the same time — the layers process all positions in parallel, a large part of why GPUs suit this work. And while the layers are identical in shape, each has its own learned weights, so each adds a different kind of refinement to the stream.

Attention

Attention deserves a closer look, because it is where the reading actually happens. The mechanism is simpler than its reputation. From its current vector, each token computes three new vectors by multiplying against three learned matrices: a query — roughly, "here is what I am looking for" — a key — "here is what I contain" — and a value — "here is what I will hand over if asked." A token's query is compared against the key of every token before it, one dot product per pair. The scores pass through a softmax, which turns them into positive weights summing to one, and the token then pulls in a mixture of the other tokens' values in exactly those proportions — heavily from tokens whose keys matched its query, faintly from everyone else. That weighted mixture is the correction added to its residual stream.

Two details complete the picture. First, causality: a token may only attend backward, to positions that come before it, never forward — the future is masked out. Second, each layer runs many of these operations side by side, in separate heads, each with its own query, key, and value matrices, and each free to learn its own notion of what is relevant to what. The weights in this figure are invented for illustration, but the patterns they imitate are routinely found in trained models: heads that simply attend to the previous word, heads that track grammatical structure, heads that resolve references. Notice what the coreference head does with "it" — most of its weight lands on "test". Nothing in the model knows what a pronoun is, and no rule about anaphora is written down anywhere; the behavior is just dot products between learned vectors that happened, during training, to make next-token prediction easier. Multiply this by dozens of heads and dozens of layers, and "reading your prompt" dissolves into these little vector comparisons repeated tens of thousands of times. When Claude Code appears to find the failing function in the file you pasted, no name is ever looked up — it is all this.

One distribution

At the top of the stack, one more multiplication. The final vector of the last token is projected against the unembedding — a matrix with one row per entry in the vocabulary — producing one score per token: the logits. A last softmax converts them into probabilities. It is worth pausing on the asymmetry here. A machine with billions of parameters has just run end to end across your whole context, and everything it bought is exactly one thing: a probability distribution over the next token. Not an answer, not a sentence. One distribution.

Something still has to turn that distribution into a choice. Always taking the single most likely token — greedy decoding — sounds sensible but produces stilted, repetitive text that easily falls into loops. Instead, the next token is sampled at random from the distribution, after reshaping it with a temperature: the logits are divided by T before the softmax. As T approaches zero the distribution sharpens toward the deterministic favorite; as it grows the distribution flattens toward noise. A companion trick, top-p, discards the long tail of barely-plausible tokens before sampling. The figure below runs the real softmax arithmetic on a set of hand-designed logits — drag the temperature slider to reshape the bars, then press Sample to watch a random draw land on the cumulative bar and pick the next token:

One token is chosen and appended to the sequence — and then the entire forward pass runs again on the slightly longer context to buy the next distribution. Every visible word of a model's reply is produced this way, one pass and one token at a time.

The KV cache

"Runs again" should worry you. To produce token N + 1, attention needs the keys and values of all N earlier tokens — and computed naively, every step would recompute all of them, even though they depend only on the unchanged prefix and come out identical every time. So they are computed once and kept: every token's keys and values, for every layer and head, are stored in accelerator memory as the KV cache. Each new step computes a query, key, and value only for the newest token, attends over the cached keys and values of everything before it, and appends its own K and V to the cache.

This one optimization shapes the economics of everything you feel as a user. The cache is large — for a big model, a few hundred kilobytes per token, so a long agent session's context runs to gigabytes — and it must be streamed out of memory on every single step, which is why generating tokens is limited by memory bandwidth rather than arithmetic, and why long contexts respond more slowly and cost more. It is also precisely the thing that prompt caching, which we met in the datacenter section, preserves between the calls of the agent loop: the conversation's keys and values stay warm so the next request need not rebuild them. In the figure below you can watch the cache grow one column per step — and flip the toggle to see what the naive world without it would recompute:

Production models refine this picture in ways we are skipping: grouped-query attention shares keys and values across heads to shrink the cache, mixture-of-experts layers route each token to a small subset of the MLP weights, and speculative decoding drafts several tokens cheaply and verifies them in a single pass. None of it changes the shape of the story told here.

Either way, at the end of the pass there now exists a token id that did not exist a moment ago — and it has to make its way back out of the datacenter to your terminal.

The Way Back

The model has committed to a token. Right now it is nothing but an integer in accelerator memory — say, id 8543 — chosen by the sampler at the end of a forward pass. For that number to become a flicker of text in your terminal, it has to make the entire journey of this article in reverse, and it starts immediately. The serving stack does not wait for a sentence, or even a word, to finish. The id is looked up in the tokenizer's vocabulary, mapped back to its bytes — perhaps Fix — and pushed onto the HTTPS connection that has been sitting open this whole time. This is the connection the CLI asked for back when it set "stream": true in the request body: instead of one response, the server answers with Content-Type: text/event-stream and keeps the socket alive, trickling the reply out as server-sent events, one small encrypted chunk at a time, back through the same TLS tunnel the request came in on.

The stream has a fixed grammar. First comes a message_start event carrying the shell of the reply — its id, the model name, and how many input tokens the prompt consumed. Then, for each block of content, a content_block_start, followed by many content_block_delta events, each holding a small fragment of text. A fragment usually corresponds to a token or a handful of them; on the wire it is just a couple of lines:

event: content_block_delta
data: {"type":"content_block_delta","index":0,
       "delta":{"type":"text_delta","text":"Fix"}}

When a block ends, a content_block_stop follows, then a message_delta with the stop_reason and the final token counts, and finally message_stop. The typewriter effect you see in the terminal is not an animation that some frontend engineer tuned to feel pleasant. It is the actual cadence of generation reaching you: each delta exists because, tens of milliseconds ago, one more decode iteration completed in a datacenter, and the result was shipped to you before the next one was even sampled. In the demonstration below, you can watch the event frames come down the wire on the left while Claude Code, on the right, accumulates their fragments into visible text. The slider changes the pace, and the button replays the stream from the top.

On your machine, the loop from the first section now runs in reverse. There, keystrokes became bytes, bytes became JSON, JSON became a request. Here, the CLI reads each event off the socket as it arrives, parses the little JSON payload, appends the fragment to its buffer, and re-renders the reply — markdown parsed live, headings and code blocks colored with the same ANSI escape sequences we met at the very beginning — dozens of times per response, once per repaint, so the text appears to grow under a blinking cursor.

Closing the loop

Very often, though, the stream does not end in prose. Instead of a text block, the model opens a content_block_start of type tool_use, naming a tool — Read, say — and then streams the tool's JSON input in input_json_delta fragments. It is worth pausing on this: the arguments to the tool call are themselves generated token by token, a JSON string assembled through the same forward passes as any sentence, and streamed back with the same event grammar. The stream then closes with stop_reason: "tool_use", and on the server, nothing further happens. The model holds no thread of execution, no pending callback. The CLI is now holding the ball.

What happens next happens entirely on your machine. Claude Code shows you the permission prompt; you press y; the CLI spawns a real process — opens the file, runs the command — against your actual disk. The model never touched it and never could; it only ever emitted text describing what it wanted. The CLI wraps the outcome in a tool_result block, appends it to the conversation, and sends the whole grown context off in a fresh POST. And with that, everything in this article happens again: the TLS handshake (skipped — the connection is still warm), the trip through the internet, the gateway, the batch, the prefill (mostly skipped — the unchanged prefix is a prompt-cache hit), the decode loop. The whole journey

Let us total up one ordinary exchange — you ask Claude Code to fix a failing test, it reads a file, edits it, and reports back. That is three or four API calls. Across them, a few hundred thousand prompt tokens pass through prefill, the great majority resolved from cache. A few thousand tokens are generated, and each one of those — every word fragment that appears under your cursor — is the product of a complete pass through hundreds of billions of weights, spread across a team of accelerators working in lockstep. All of it, typically, inside a minute. It is tempting, watching text compose itself in a terminal, to file the whole thing under magic and move on. But nothing in the path we have traced is magic. It is escape codes and JSON, sockets and certificates, routers and load balancers, matrices multiplied very quickly, and a sampler picking one integer at a time — each layer ordinary on its own, each one inspectable if you care to look. The depth of the stack is real, and so is the plainness of every step in it. The next time a reply types itself out in front of you, you will know that the small pause before each word is not a flourish. It is the round trip — the whole round trip — happening again.

Every figure on this page is a simplification — real systems have more layers, more caches, and more failure handling than any diagram can hold. But the shape of the machinery is faithful: a terminal, a JSON document, a fleet of GPUs, and a model predicting one token at a time, in a loop.
